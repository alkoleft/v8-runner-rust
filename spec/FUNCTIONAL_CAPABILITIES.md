# Функциональные возможности проекта

> Примечание: это внутренний инвентаризационный документ по текущему поведению системы. Публичная пользовательская документация теперь находится в `README.md`, `docs/CAPABILITIES.md` и `docs/DEEP_DIVE.md`. Внутри могут встречаться исторические или переходные пояснения, полезные для maintainers, но не предназначенные как внешний продуктовый reference.

Документ изначально собирался по актуальному коду проекта, а не по старым описаниям в `README.md`, `docs/FEATURES.md` и `docker-compose.yml`; при расхождениях с текущим публичным doc-layer источником истины для пользователей считаются `README.md`, `docs/CAPABILITIES.md` и `docs/DEEP_DIVE.md`.

Цель документа: зафиксировать, что продукт реально делает сейчас, какие задачи он решает, в каком порядке выполняет операции и какие технические особенности нельзя потерять при переписывании на другой стек.

## 1. Назначение продукта

`v8-runner` это MCP-сервер, который дает AI-ассистенту управляемый доступ к типовым операциям над проектом 1С:

- загрузка исходников в информационную базу;
- конвертация EDT-проекта в Designer-формат;
- запуск тестов YaXUnit;
- выгрузка конфигурации и расширений из ИБ обратно в файлы;
- синтаксические проверки через EDT и Конфигуратор;
- запуск приложений 1С.

По сути это прослойка между MCP-клиентом и локальными утилитами 1С (`1cv8`, `1cv8c`, `ibcmd`, `1cedtcli`).

## 2. Какие задачи проект решает

### 2.1. Автоматизация цикла "изменил код -> загрузил в ИБ -> прогнал тесты"

Проект убирает ручные шаги, которые обычно делает разработчик:

- понять, изменились ли исходники;
- если проект в EDT, сначала экспортировать измененные проекты в Designer-формат;
- загрузить изменения в ИБ;
- при возможности не грузить весь проект целиком, а выполнить частичную загрузку;
- после успешной сборки запустить YaXUnit и вернуть структурированный отчет.

Практический эффект:

- ускоряет локальный цикл проверки гипотез;
- позволяет запускать проверки прямо из AI-ассистента;
- снижает количество "забыл сначала обновить базу, а потом уже запускать тесты".

### 2.2. Поддержка двух форматов исходников

Проект умеет работать с двумя вариантами хранения исходников:

- `DESIGNER` — файлы уже лежат в формате, пригодном для загрузки в ИБ;
- `EDT` — исходники лежат как проекты EDT, перед загрузкой в ИБ их нужно экспортировать через `1cedtcli`.

Практический эффект:

- один и тот же MCP-интерфейс работает и для "классических" проектов, и для проектов в EDT;
- для EDT не требуется ручной промежуточный экспорт перед сборкой.

### 2.3. Инкрементальная обработка вместо "всегда полный rebuild"

Перед конвертацией и/или сборкой проект анализирует изменения файлов и решает:

- можно ли пропустить шаг полностью;
- можно ли обработать только затронутые `source-set`;
- можно ли выполнить частичную загрузку вместо полной.

Практический эффект:

- меньше время ожидания;
- меньше лишних обращений к платформе 1С;
- лучше подходит для больших конфигураций и проектов с несколькими расширениями.

### 2.4. Автоматизация обратной синхронизации "ИБ -> файлы"

Проект умеет выгружать конфигурацию из информационной базы обратно в файловую структуру проекта.

Это закрывает сценарии:

- разработчик или аналитик внес изменения в Конфигураторе;
- нужно вернуть эти изменения в файловые исходники;
- нужно выгрузить только одно расширение;
- нужно точечно выгрузить отдельные объекты.

### 2.5. Контроль качества без запуска тестов

Проект умеет выполнять синтаксические проверки:

- через `1C:EDT validate`;
- через `CheckConfig` Конфигуратора;
- через `CheckModules` Конфигуратора.

Это закрывает сценарии:

- быстро проверить проект до запуска тестов;
- получить структурированный список ошибок/предупреждений;
- использовать проект как quality gate в ассистенте или CI-подобном сценарии.

### 2.6. Управление локальным окружением 1С

Проект умеет запускать приложения 1С:

- Конфигуратор;
- тонкий клиент;
- толстый клиент.

Это не бизнес-функция в узком смысле, но это важная часть рабочего процесса: ассистент может не только запускать проверки, но и открыть нужный инструмент 1С по команде.

## 3. Реально доступные MCP-инструменты

На текущий момент в коде реализованы 8 tool-методов:

| Tool | Что делает | Важные особенности |
| --- | --- | --- |
| `run_all_tests` | Запускает все тесты YaXUnit | Перед тестами всегда вызывает сборку |
| `run_module_tests` | Запускает тесты указанного модуля | Перед тестами всегда вызывает сборку |
| `build_project` | Загружает исходники в ИБ | Поддерживает `fullRebuild`, но это именно сброс кэша изменений |
| `dump_config` | Выгружает конфигурацию/расширение из ИБ в файлы | Режимы `FULL`, `INCREMENTAL`, `PARTIAL` |
| `launch_app` | Запускает Designer / thin / thick client | Возвращает статус запуска |
| `check_syntax_edt` | Запускает `1cedtcli validate` | Работает по именам EDT-проектов |
| `check_syntax_designer_config` | Запускает `CheckConfig` | Возвращает структурированные issues |
| `check_syntax_designer_modules` | Запускает `CheckModules` | Возвращает структурированные issues |

Важно:

- в текущем коде нет MCP-tool для `list_modules`;
- в текущем коде нет MCP-tool для `get_configuration`;
- в текущем коде нет MCP-tool для `check_platform`.

Эти пункты упоминаются в старых документах, но в коде отсутствуют.

## 4. Порядок работы системы

### 4.1. Что происходит при старте приложения

1. Spring Boot поднимает приложение в безвебовом режиме.
2. Загружается внешний `v8project.yaml` через `SPRING_CONFIG_IMPORT`, если он передан.
3. Валидируется конфигурация проекта:
   - существует ли `basePath`;
   - задан ли хотя бы один `CONFIGURATION` в `source-set`;
   - валидна ли строка подключения;
   - допустимы ли `format`, `builder` и прочие опции.
4. Если проект в формате `EDT` и включен `app.tools.edt-cli.auto-start`, запускается фоновый старт `1cedtcli`.

### 4.2. Фоновая работа EDT CLI

Если включен автозапуск EDT CLI:

- поднимается отдельный интерактивный процесс `1cedtcli`;
- процесс инициализируется до появления prompt `1C:EDT>`;
- сохраняется ссылка на готовый `InteractiveProcessExecutor`;
- запускается фоновый монитор процесса;
- монитор раз в 5 секунд проверяет, что процесс жив и отвечает;
- при сбое выполняется автоматический перезапуск.

Критично:

- это единственный реально существующий фоновый процесс уровня приложения;
- отдельного фонового демона для отслеживания изменений файлов нет.

### 4.3. Как определяется наличие изменений файлов

Изменения ищутся не в фоне, а по запросу, непосредственно перед конвертацией/сборкой.

Алгоритм двухфазный:

1. Быстрый проход по `lastModified` файлов.
2. Для кандидатов вычисляется хеш содержимого.
3. Изменения группируются по `source-set`.
4. После успешной обработки обновляются хеши и timestamp.

Игнорируются служебные каталоги и файлы:

- `.git/`
- `.gradle/`
- `build/`
- `target/`
- `temp/`
- `tmp/`
- `ConfigDumpInfo.xml`
- `.yaxunit/`

### 4.4. Порядок работы `build_project`

### Для формата `DESIGNER`

1. Опционально очищается кэш изменений, если вызван `fullRebuild=true`.
2. Выполняется анализ файловых изменений в текущем Designer-source-set.
3. Если изменений нет, обновление ИБ пропускается.
4. Если изменения есть, выбираются только затронутые `source-set`.
5. Для каждого `source-set` выбирается режим загрузки:
   - частичная загрузка, если измененных файлов мало и не затронут `Configuration.xml`;
   - полная загрузка в остальных случаях.
6. Выполняется загрузка основной конфигурации и расширений.
7. После успешной загрузки сохраняется новое состояние хешей и timestamp.

### Для формата `EDT`

Пайплайн двухступенчатый:

1. Анализируются изменения в EDT-source-set.
2. Если изменений нет, конвертация `EDT -> Designer` пропускается.
3. Если изменения есть:
   - выбираются только измененные `source-set`;
   - каждый из них экспортируется через `1cedtcli export`;
   - экспорт идет во временный Designer-каталог `workPath`.
4. После конвертации выполняется обычный Designer-пайплайн сборки:
   - анализируются изменения уже в Designer-представлении;
   - в ИБ загружаются только затронутые `source-set`.

Таким образом, у EDT-проекта есть две независимые стадии отслеживания изменений:

- изменения EDT-исходников;
- изменения сгенерированных Designer-исходников.

### 4.5. Порядок работы `run_all_tests` и `run_module_tests`

1. Опционально очищаются файлы логов, если выставлен `CLEAN_BEFORE_EXECUTION=true`.
2. Всегда выполняется `build()`.
3. Если сборка неуспешна, тесты не стартуют.
4. Формируется временный JSON-конфиг YaXUnit:
   - фильтр по модулям или без фильтра;
   - путь к JUnit-отчету;
   - путь к YaXUnit-логу.
5. Через `1С:Предприятие` запускается команда `RunUnitTests=<configPath>`.
6. После завершения:
   - читается JUnit XML;
   - собирается общая статистика;
   - из YaXUnit-лога вытаскиваются блоки `[ERR]`;
   - итог возвращается в структурированном MCP-ответе.

Особенность:

- тестовый прогон идет через `EnterpriseDsl`, который по умолчанию использует `THIN_CLIENT` (`1cv8c`).

### 4.6. Порядок работы `dump_config`

1. Определяется целевой `source-set` для выгрузки.
2. Если передано `extension`, выбирается каталог соответствующего расширения.
3. Выбирается режим:
   - `FULL`;
   - `INCREMENTAL`;
   - `PARTIAL`.
4. Выполняется выгрузка через выбранный backend:
   - `DESIGNER`;
   - `IBCMD`.

Практический смысл режимов:

- `FULL` — полная выгрузка;
- `INCREMENTAL` — только изменения относительно `ConfigDumpInfo.xml` или эквивалентной логики backend;
- `PARTIAL` — точечная выгрузка объектов по списку.

Дополнение по backend semantics:

- для `DESIGNER` `PARTIAL` использует `DumpConfigToFiles -partial -listFile`;
- для `IBCMD` объектная partial-выгрузка напрямую не поддержана, поэтому `PARTIAL`
  деградирует в incremental export по resolved target с warning, но в ответе сохраняется
  requested mode `PARTIAL`.

### 4.7. Порядок работы синтаксических проверок

### `check_syntax_edt`

1. Выбираются имена проектов EDT:
   - либо явный `projectName`;
   - либо все `source-set` с `purpose=MAIN`.
2. Для каждого проекта запускается `1cedtcli validate`.
3. Лог проверки пишется в файл.
4. TSV-подобный вывод EDT разбирается в структурированный список `issues`.

### `check_syntax_designer_config`

1. Формируется команда `CheckConfig`.
2. Включаются режимы проверки по параметрам tool.
3. Запускается Конфигуратор.
4. Лог разбирается на:
   - ошибки объектов;
   - ошибки модулей с координатами строк/колонок.

### `check_syntax_designer_modules`

1. Формируется команда `CheckModules`.
2. Обязательно должен быть включен хотя бы один режим проверки.
3. Лог разбирается тем же парсером ошибок Конфигуратора.

### 4.8. Порядок работы `launch_app`

1. Псевдоним приводится к одному из типов:
   - `DESIGNER`
   - `THIN_CLIENT`
   - `THICK_CLIENT`
2. Подбирается нужная утилита платформы.
3. Команда запускается асинхронно.
4. В ответ возвращается факт запуска и PID процесса.

## 5. Что важно не потерять при переписывании

### 5.1. Отдельного фонового мониторинга файлов нет

Сейчас система не использует `WatchService`, inotify, fs events и т.д.

Фактическая модель такая:

- нет постоянного file watcher;
- есть on-demand scan перед build/convert;
- есть отдельный фоновый монитор только у процесса `1cedtcli`.

Если переписать систему и случайно добавить "реальный watcher" как обязательный компонент, это будет уже другая операционная модель.

### 5.2. Состояние изменений хранится на диске

Для инкрементальной логики проект хранит состояние между запусками:

- хеши файлов;
- timestamp последней успешной обработки source-set.

Путь хранения:

- `${java.io.tmpdir}/mcp-yaxunit-runner/<app.id или hash(basePath)>/hash-storages/*.redb`

Важно:

- в коде хранилище фактически `redb`-based;
- старые документы все еще могут называть это MapDB/JSON.

### 5.3. Для EDT есть рабочая копия Designer-исходников

Для формата `EDT` проект создает отдельное рабочее пространство Designer-исходников в `workPath`.

Зачем это нужно:

- не грузить EDT-проекты напрямую в ИБ;
- иметь промежуточный файловый слой после `1cedtcli export`;
- отдельно отслеживать изменения EDT и уже сконвертированных Designer-файлов.

Это один из самых критичных архитектурных моментов проекта.

### 5.4. Частичная загрузка устроена не "по объектам", а "по файлам"

Частичная загрузка в build-пайплайне строится так:

- для набора измененных файлов генерируется `listFile`;
- если изменен `.bsl`, в список добавляются связанные XML и весь каталог объекта;
- если затронут `Configuration.xml`, система принудительно уходит в полную загрузку;
- порог переключения задается через `BuildProperties.partialLoadThreshold` (по умолчанию `20`).

Это не просто "инкрементальная сборка", а очень конкретная файловая стратегия.

### 5.5. `fullRebuild` это сброс кэша, а не отдельный режим полной сборки платформы

При `build_project(fullRebuild=true)` проект делает следующее:

- очищает кэш хешей для всех source-set;
- после этого вся логика повторно считает все файлы измененными.

То есть это не отдельный backend-режим команды 1С, а форсированный пересчет изменений.

### 5.6. Ограничение IBCMD-backend

Текущая реализация `IBCMD` опирается на извлечение `File=` из строки подключения.

Следствие:

- backend `IBCMD` в текущем коде ориентирован именно на файловую базу;
- серверная строка подключения в `IbcmdDsl.connect` фактически не поддержана.

Это критичное ограничение, которое легко потерять из виду, если смотреть только на docs.

### 5.7. Ограничения выгрузки через IBCMD

В `IbcmdDumpAction`:

- `FULL` реализован честно через `--force`;
- `INCREMENTAL` идет через `--sync`;
- `PARTIAL` по списку объектов напрямую не поддержан и сейчас деградирует в `--sync`
  для resolved configuration/extension target с явным warning.

Если переписывать на другой стек, надо заранее решить:

- сохраняем эту же семантику;
- или делаем настоящее object-level partial dump.

### 5.8. Старые docs содержат устаревшие сведения

В актуальном коде есть расхождения с документацией:

- нет `list_modules`, `get_configuration`, `check_platform`;
- нет отдельной конфигурационной опции `interactive-mode`, хотя она описана в docs;
- хранилище изменений уже не MapDB/JSON, а `redb`;
- `docker-compose.yml` описывает функциональность, которой в приложении сейчас нет.

Для переписывания источником истины нужно считать `src/main/kotlin`, а не старые документы.

## 6. Минимальный функциональный инвариант для переписывания

Если переносить проект на другой стек, минимально нужно сохранить:

1. MCP-слой с текущими 8 tool-операциями.
2. Двухступенчатый пайплайн для EDT: `change detection -> export -> build`.
3. On-demand анализ изменений с персистентным кэшем.
4. Частичную загрузку по файлам с fallback на полную.
5. Автоматическую сборку перед запуском тестов.
6. Фоновый интерактивный `1cedtcli` с мониторингом и перезапуском.
7. Парсинг JUnit-отчета YaXUnit и ошибок из логов.
8. Структурированные результаты проверок и тестов, пригодные для MCP-клиента.
