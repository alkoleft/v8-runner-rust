# ADR-0001: Границы поддержки IBCMD как ограниченного backend

- Статус: `accepted`
- Дата: `2026-04-02`

## Контекст

В проекте `IBCMD` уже используется как альтернативный backend, но публичный контракт был размазан между кодом и несколькими документами.
Это повышало риск дрейфа формулировок и неоднозначной трактовки поддержки.

Нужно зафиксировать единую границу: какие сценарии поддерживаются, какие ограничения считаются частью продукта, а не временным пробелом.

## Решение

Принять `IBCMD` как **ограниченный backend** со следующими границами:

1. Поддерживаемые сценарии: `init`, `build`, `dump`, `extensions`.
2. Подключение к ИБ: только файловая информационная база.
3. `dump --mode partial`:
- object-scoped partial dump нативно не поддержан;
- запрос `partial` деградирует в `incremental export` с предупреждением;
- в результирующем payload сохраняется запрошенный режим `PARTIAL`.
4. `IBCMD` не считается полной заменой `DESIGNER` и не обязан повторять всю его функциональность.

## Неграницы (Non-goals)

1. Добавление object-scoped partial dump для `IBCMD`.
2. Поддержка server-based connection для `IBCMD`.
3. Достижение feature parity `IBCMD` и `DESIGNER`.

## Последствия

1. Публичная документация должна описывать `IBCMD` как отдельный ограниченный контракт.
2. Внутренние и публичные документы должны обновляться синхронно при изменении границ.
3. План дальнейших работ по поддержке `IBCMD` выполняется отдельно (тестовое покрытие, live smoke, диагностика), без расширения текущего контракта.
