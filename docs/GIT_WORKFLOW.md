# Git-Ориентированный Локальный Workflow

Этот документ фиксирует, какие команды `v8-test-runner` действительно нужны для повседневной локальной разработки в проекте, где исходники живут в Git, а информационная база используется как исполняемая среда и промежуточный runtime.

Цель не в том, чтобы перечислить все поддержанные команды, а в том, чтобы выбрать минимальный рабочий набор для сценария:

`git checkout / edit / build / validate / test / dump / commit`

Для `IBCMD` в этом workflow действует зафиксированная граница из [ADR-0001](decisions/0001-granitsy-podderzhki-ibcmd-kak-ogranichennogo-backend.md): backend ограничен сценариями `init`, `build`, `dump`, `extensions`, поддерживает только файловую ИБ, а `dump --mode partial` деградирует в incremental export с предупреждением.

## Основной набор команд

### Команды ежедневного цикла

- `build`
- `syntax`
- `test`
- `dump`

### Команды поддержки workflow

- `launch`
- `extensions`
- `init`

## Почему выбран именно этот набор

### `build`

Это центральная команда локальной разработки.

Она отвечает за переход из состояния "в Git изменились файлы" в состояние "изменения реально применены в информационной базе". Без нее остальные команды либо бессмысленны, либо работают по устаревшему состоянию ИБ.

На уровне низкоуровневых платформенных операций она опирается в первую очередь на:

- Designer backend: `/LoadConfigFromFiles` + `/UpdateDBCfg`
- Designer partial path: `/LoadConfigFromFiles -partial -listFile ...` + `/UpdateDBCfg`
- IBCMD backend: `config import` + `config apply`

### `syntax`

Это быстрый quality gate до запуска тестов.

Команда полезна в git-цикле потому, что дешевле тестов и позволяет рано отсеивать синтаксические и платформенные ошибки.

На уровне низкоуровневых операций:

- `syntax designer-config` -> `/CheckConfig`
- `syntax designer-modules` -> `/CheckModules`
- `syntax edt` -> `1cedtcli validate`

### `test`

Это сценарий подтверждения поведения, а не просто запуск YaXUnit.

Для git-ориентированной разработки важно, что `test` всегда включает предварительный `build`, то есть агенту или разработчику не нужно держать в голове отдельный шаг синхронизации ИБ.

Низкоуровневый смысл:

- сначала `build`
- затем запуск `Enterprise` с YaXUnit-конфигом

### `dump`

Это обратная синхронизация из ИБ в Git-исходники.

Команда обязательна для нормальной разработки с Git, потому что изменения могут появляться не только в файловых исходниках, но и в самой ИБ: ручная правка в Конфигураторе, быстрый hotfix, анализ проблемы на живой базе, исправление формы или модуля вручную.

На уровне Designer backend:

- `/DumpConfigToFiles`
- при partial path: `/DumpConfigToFiles -partial -listFile ...`
- при `builder=IBCMD` object-scoped partial dump не поддерживается, поэтому `partial` работает как degraded incremental export с предупреждением

### `launch`

Это не основная команда сборки, но важная operational-команда локального цикла.

Она нужна, когда после `build` и `syntax` требуется открыть Designer, тонкий или толстый клиент для ручной проверки, воспроизведения бага или интерактивной отладки.

### `extensions`

Это специализированная, но полезная команда для git-проекта с расширениями.

Она нужна не вместо `build`, а как более узкий восстановительный или подготовительный сценарий, когда нужно синхронизировать свойства расширений без полной загрузки исходников.

### `init`

Это не ежедневная команда, но базовая команда жизненного цикла репозитория.

Она нужна при первом клонировании проекта, пересоздании рабочей базы или инициализации EDT workspace.

## Команды, которые не входят в основной git-cycle

Следующие команды и классы операций не считаются основными для повседневной разработки с Git:

- `mcp serve`: это способ публикации automation surface, а не шаг локального цикла разработки
- низкоуровневые repository-команды 1С: в Git-centric модели основное хранилище истины — Git, а не хранилище конфигурации 1С
- команды администрирования ИБ вроде восстановления, очистки, integrity repair: это эксплуатационные или аварийные сценарии, а не обычная разработка
- команды поставки и дистрибутивов: это release/distribution path, а не локальный inner loop

## Рекомендуемые сценарии

### Сценарий 1. Обычный цикл после правки файлов

Когда использовать:

- изменены исходники в рабочем дереве
- нужно быстро проверить, что ИБ синхронизирована с Git-состоянием

Последовательность:

1. `build`
2. при необходимости `launch --mode designer|thin|thick`

Почему:

- `build` уже умеет выбирать incremental/partial/full путь
- это основной inner loop вместо ручного запуска Designer batch-команд

### Сценарий 2. Быстрая проверка перед commit

Когда использовать:

- изменения локальны и ожидается быстрый feedback
- хочется отловить ошибки раньше, чем запускать полный тестовый прогон

Последовательность:

1. `build`
2. `syntax designer-modules` или `syntax edt`
3. при необходимости `syntax designer-config`

Почему:

- `build` приводит ИБ в актуальное состояние
- `syntax` дешевле тестов и хорошо работает как pre-commit gate

### Сценарий 3. Проверка перед push или PR

Когда использовать:

- локальная ветка готовится к отправке
- нужна более сильная уверенность, чем просто синтаксическая проверка

Последовательность:

1. `test all`

Почему:

- `test` уже включает обязательный `build`
- это основной high-confidence сценарий перед публикацией изменений в Git

### Сценарий 4. Прицельная проверка после локального изменения

Когда использовать:

- изменен один модуль или узкая область
- полный прогон тестов слишком дорог

Последовательность:

1. `test module <MODULE_NAME>`

Почему:

- команда включает `build`
- хорошо подходит для короткого цикла "поправил -> проверил конкретный модуль"

### Сценарий 5. Переключение ветки, rebase или массовое изменение структуры

Когда использовать:

- был `git checkout` на сильно отличающуюся ветку
- был `rebase`, массовое перемещение файлов, regen EDT/Designer артефактов
- локальное состояние change detection больше нельзя считать надежным

Последовательность:

1. `build --full-rebuild`
2. при необходимости `test all`

Почему:

- это безопасный способ пересинхронизировать ИБ после крупных изменений Git-дерева
- `--full-rebuild` нужен именно для сброса incremental assumptions

### Сценарий 6. Возврат изменений из ИБ обратно в Git

Когда использовать:

- изменения были сделаны вручную в Конфигураторе
- нужно зафиксировать runtime-изменения обратно в файловые исходники

Последовательность:

1. `dump --mode incremental`
2. при необходимости `dump --mode partial --object <TYPE:NAME>`
3. `git diff`

Почему:

- `dump` превращает текущее состояние ИБ обратно в Git-visible изменения
- после этого можно продолжить обычный git workflow: review, commit, push

### Сценарий 7. Подготовка или восстановление расширений

Когда использовать:

- проект активно использует расширения
- нужно быстро выровнять свойства расширения без полной сборки

Последовательность:

1. `extensions [--name <SOURCE_SET>...]`
2. затем обычный `build`

Почему:

- `extensions` полезен как узкий preparatory/fixup шаг
- но не заменяет основной `build`-pipeline

### Сценарий 8. Новый клон репозитория или пересоздание среды

Когда использовать:

- первый запуск после `git clone`
- рабочая база удалена
- нужно заново поднять локальное окружение

Последовательность:

1. `init`
2. `build`
3. `syntax`
4. `test all`

Почему:

- `init` поднимает минимальную рабочую среду
- остальные шаги подтверждают, что проект не только инициализирован, но и реально собирается и запускается

## Практическая рекомендация по умолчанию

Если нужен минимальный набор команд для "нормальной локальной разработки с Git", его стоит считать таким:

1. `build` как центральная команда синхронизации Git -> ИБ
2. `syntax` как быстрый gate
3. `test` как сценарий подтверждения поведения
4. `dump` как обратная синхронизация ИБ -> Git

`launch`, `extensions` и `init` стоит считать не ядром inner loop, а необходимыми supporting-командами вокруг него.
